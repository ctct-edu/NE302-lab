# Session 1: 要件の言語化と骨格設計

## 前提

- VS Code または Cursor がインストールされていること
- `exercise/` フォルダを作業フォルダにコピーしていること

---

## この演習で使う AI 機能

この演習では、主に **チャット（Chat パネル）** を使用します。AI と対話しながら要件や設計を整理します。

チャットパネルの開き方：

| ツール         | 操作                                                                              |
| -------------- | --------------------------------------------------------------------------------- |
| GitHub Copilot | サイドバーのチャットアイコン、または `Ctrl + Shift + I`（Mac: `Cmd + Shift + I`） |
| Cursor         | `Ctrl + L`（Mac: `Cmd + L`）                                                      |

---

## exercise フォルダの内容

`exercise/` フォルダにはプロジェクトの雛形が含まれています：

| ファイル/フォルダ       | 内容                                |
| ----------------------- | ----------------------------------- |
| `Inventory.sln`         | ソリューションファイル              |

## 演習 1: Cursor Rules / Copilot 設定の準備（5 分）

プロジェクト固有のルールを AI に伝える設定ファイルを作成します。

使用する AI 機能: **Chat**

**Cursor の場合：**

`.cursor/rules/general.mdc` を作成します。

```sh
mkdir -p .cursor/rules
```

```markdown
# プロジェクトルール

## 技術スタック

- .NET 8
- ASP.NET Core Web API
- Entity Framework Core
- MySQL
- xUnit（ユニットテスト）
- Playwright（E2E テスト）

## 命名規則

- ファイル名: PascalCase（例: ProductService.cs）
- クラス名: PascalCase（例: ProductService）
- 変数名: camelCase（例: productName）
- プロパティ名: PascalCase（例: ProductName）
- 名前空間: PascalCase（例: Inventory.Services）
```

**GitHub Copilot の場合：**

`.github/copilot-instructions.md` を作成します。

```sh
mkdir -p .github
```

## 演習 2: ユースケースの洗い出し（20 分）

AI を活用して在庫管理システムの要件を言語化し、ユースケースを整理します。

### ステップ 1: AI と対話して要件を整理する

使用する AI 機能: **Chat**

チャットパネルを開き、以下のプロンプトを入力します。

**入力するプロンプト：**

```
在庫管理システムを作りたいです。
小規模な店舗向けで、以下の機能が必要です。

- 商品の登録・検索・一覧表示
- 在庫数の管理（入庫・出庫）
- 在庫が少なくなったら警告

これを要件定義書としてまとめてください。
MUST（必須）と WANT（あれば良い）に分類してください。
```

期待される AI の出力例：

| 分類 | 機能         | 説明                         |
| ---- | ------------ | ---------------------------- |
| MUST | 商品登録     | 商品名、カテゴリ、単価を登録 |
| MUST | 商品一覧     | 登録済み商品の一覧表示       |
| MUST | 在庫管理     | 入庫・出庫の記録             |
| MUST | 在庫アラート | 閾値以下で警告表示           |
| WANT | 検索機能     | 商品名、カテゴリで絞り込み   |
| WANT | レポート出力 | 在庫状況の CSV 出力          |

> **この講座では、MUST 機能をすべて実装します。** Session 3-4 で商品登録・検索・在庫管理のテストと実装を、Session 5 で在庫アラート・発注管理を実装します。WANT 機能は実装しません。

### ステップ 2: ユースケースを洗い出す

使用する AI 機能: **Chat**

以下のプロンプトを入力します。

**入力するプロンプト：**

```
在庫管理システムのユースケースを洗い出してください。
各ユースケースについて、以下の情報を含めてください。

- アクター（誰が操作するか）
- 事前条件
- 基本フロー
- 代替フロー（エラー時など）
```

期待される出力例（一部）：

```
【ユースケース】商品を登録する

アクター: 店長
事前条件: ログイン済み
基本フロー:
  1. 商品登録画面を開く
  2. 商品名、カテゴリ、単価を入力
  3. 登録ボタンを押す
  4. 登録完了メッセージが表示される
代替フロー:
  - 商品名が空の場合 → エラーメッセージを表示
  - 単価が負数の場合 → エラーメッセージを表示
```

## 演習 3: システム構成の設計（20 分）

技術スタックとレイヤー構成を決めます。

### ステップ 1: 技術スタックを決める

使用する AI 機能: **Chat**

以下のプロンプトを入力します。

**入力するプロンプト：**

```
在庫管理システムを C# で作ります。
以下の技術スタックを使う予定です。

- .NET 8
- ASP.NET Core Web API（Web フレームワーク）
- Entity Framework Core（ORM）
- MySQL（データベース）
- xUnit（テスト）

このスタックで問題がないか確認してください。
また、ディレクトリ構成の提案をしてください。
```

各技術の役割：

| 技術                  | 役割                                           |
| --------------------- | ---------------------------------------------- |
| .NET 8                | プラットフォーム（LTS バージョン）             |
| ASP.NET Core Web API  | C# の Web フレームワーク                       |
| Entity Framework Core | ORM（オブジェクトとデータベースのマッピング）  |
| xUnit                 | .NET の標準テストフレームワーク                |

### ステップ 2: レイヤー構成を設計する

使用する AI 機能: **Chat**

以下のプロンプトを入力します。

**入力するプロンプト：**

```
以下のレイヤー構成で在庫管理システムを設計します。

- Controller: HTTP リクエストの受付
- Service: ビジネスロジック
- Repository: データベースアクセス
- Entity: データ構造の定義

この構成で、商品登録の処理フローを説明してください。
```

各レイヤーの役割：

| レイヤー   | 責務                                          | 例                      |
| ---------- | --------------------------------------------- | ----------------------- |
| Controller | HTTP リクエストを受け取り、Service を呼ぶ     | `POST /api/products`    |
| Service    | ビジネスロジック（検証、計算など）            | 在庫が足りるか確認      |
| Repository | データベースとのやり取り                      | INSERT, SELECT 文の実行 |
| Entity     | データの型定義（EF Core エンティティ）        | Product, Inventory      |

> **Note**: ASP.NET Core では一般的に「Routes」ではなく「Controller」、「Models」ではなく「Entity」という用語を使います。

## 演習 4: 要件定義書の作成（15 分）

AI の出力を元に、要件定義書と設計書を Markdown ファイルとして保存します。

```sh
mkdir docs
```

- AI に要件定義書を生成させ、内容をレビュー・修正して `docs/requirements.md` として保存します。
- AI にシステム構成の設計書を生成させ、内容をレビュー・修正して `docs/design.md` として保存します。

完成したら、`Ans/` フォルダの完成例と見比べて確認してください。
