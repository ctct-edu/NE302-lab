# Session 1: 要件の言語化と骨格設計

## 前提

- VS Code または Cursor がインストールされていること
- `exercise/` フォルダを作業フォルダにコピーしていること

---

## この演習で使う AI 機能

この演習では、主に **チャット（Chat パネル）** を使用します。AI と対話しながら要件や設計を整理します。

チャットパネルの開き方：

| ツール         | 操作                                                                              |
| -------------- | --------------------------------------------------------------------------------- |
| GitHub Copilot | サイドバーのチャットアイコン、または `Ctrl + Shift + I`（Mac: `Cmd + Shift + I`） |
| Cursor         | `Ctrl + L`（Mac: `Cmd + L`）                                                      |

---

## exercise フォルダの内容

`exercise/` フォルダにはプロジェクトの雛形が含まれています：

| ファイル/フォルダ       | 内容                                |
| ----------------------- | ----------------------------------- |
| `requirements.txt`      | Python パッケージ依存関係           |
| `manage.py`             | Django 管理コマンド                 |
| `inventory/`            | Django アプリケーション             |
| `config/`               | Django プロジェクト設定             |
| `docs/`                 | ドキュメント用ディレクトリ          |

## 演習 1: Cursor Rules / Copilot 設定の準備（5 分）

プロジェクト固有のルールを AI に伝える設定ファイルを作成します。

使用する AI 機能: **Chat**

**Cursor の場合：**

`.cursor/rules/general.mdc` を作成します。

```sh
mkdir -p .cursor/rules
```

```markdown
# プロジェクトルール

## 技術スタック

- Python 3.12
- Django 5.x
- Django REST Framework
- MySQL
- Node.js（Playwright 実行用）
- pytest（ユニットテスト）
- Playwright（TypeScript / E2E テスト）

## 命名規則

- ファイル名: snake_case（例: product_service.py）
- クラス名: PascalCase（例: ProductService）
- 変数名: snake_case（例: product_name）
- 関数名: snake_case（例: get_products）
```

**GitHub Copilot の場合：**

`.github/copilot-instructions.md` を作成します。

```sh
mkdir -p .github
```

## 演習 2: ユースケースの洗い出し（20 分）

AI を活用して在庫管理システムの要件を言語化し、ユースケースを整理します。

### ステップ 1: AI と対話して要件を整理する

使用する AI 機能: **Chat**

チャットパネルを開き、以下のプロンプトを入力します。

**入力するプロンプト：**

```
在庫管理システムを作りたいです。
小規模な店舗向けで、以下の機能が必要です。

- 商品の登録・検索・一覧表示
- 在庫数の管理（入庫・出庫）
- 在庫が少なくなったら警告

これを要件定義書としてまとめてください。
MUST（必須）と WANT（あれば良い）に分類してください。
```

期待される AI の出力例：

| 分類 | 機能         | 説明                         |
| ---- | ------------ | ---------------------------- |
| MUST | 商品登録     | 商品名、カテゴリ、単価を登録 |
| MUST | 商品一覧     | 登録済み商品の一覧表示       |
| MUST | 在庫管理     | 入庫・出庫の記録             |
| MUST | 在庫アラート | 閾値以下で警告表示           |
| WANT | 検索機能     | 商品名、カテゴリで絞り込み   |
| WANT | レポート出力 | 在庫状況の CSV 出力          |

> **この講座では、MUST 機能をすべて実装します。** Session 3-4 で商品登録・検索・在庫管理のテストと実装を、Session 5 で在庫アラート・発注管理を実装します。WANT 機能は実装しません。

### ステップ 2: ユースケースを洗い出す

使用する AI 機能: **Chat**

以下のプロンプトを入力します。

**入力するプロンプト：**

```
在庫管理システムのユースケースを洗い出してください。
各ユースケースについて、以下の情報を含めてください。

- アクター（誰が操作するか）
- 事前条件
- 基本フロー
- 代替フロー（エラー時など）
```

期待される出力例（一部）：

```
【ユースケース】商品を登録する

アクター: 店長
事前条件: ログイン済み
基本フロー:
  1. 商品登録画面を開く
  2. 商品名、カテゴリ、単価を入力
  3. 登録ボタンを押す
  4. 登録完了メッセージが表示される
代替フロー:
  - 商品名が空の場合 → エラーメッセージを表示
  - 単価が負数の場合 → エラーメッセージを表示
```

## 演習 3: システム構成の設計（20 分）

技術スタックとレイヤー構成を決めます。

### ステップ 1: 技術スタックを決める

使用する AI 機能: **Chat**

以下のプロンプトを入力します。

**入力するプロンプト：**

```
在庫管理システムを Python で作ります。
以下の技術スタックを使う予定です。

- Python 3.12
- Django 5.x（Web フレームワーク）
- Django REST Framework（API 構築）
- MySQL（データベース）
- Node.js 20.x（Playwright 実行）
- pytest（テスト）
- Playwright（TypeScript / E2E テスト）

このスタックで問題がないか確認してください。
また、ディレクトリ構成の提案をしてください。
```

各技術の役割：

| 技術                   | 役割                                           |
| ---------------------- | ---------------------------------------------- |
| Python 3.12            | プログラミング言語（最新安定版）               |
| Django                 | Python の Web フレームワーク                   |
| Django REST Framework  | RESTful API を簡単に構築するためのツールキット |
| Node.js                | Playwright（TypeScript）の実行環境             |
| pytest                 | Python の柔軟なテストフレームワーク            |
| Playwright             | E2E テストフレームワーク（TypeScript で実装）   |

### ステップ 2: レイヤー構成を設計する

使用する AI 機能: **Chat**

以下のプロンプトを入力します。

**入力するプロンプト：**

```
以下のレイヤー構成で在庫管理システムを設計します。

- View: HTTP リクエストの受付（API エンドポイント）
- Service: ビジネスロジック
- Model: データ構造の定義（Django ORM）
- Serializer: データの検証と変換

この構成で、商品登録の処理フローを説明してください。
```

各レイヤーの役割：

| レイヤー   | 責務                                          | 例                      |
| ---------- | --------------------------------------------- | ----------------------- |
| View       | HTTP リクエストを受け取り、Service を呼ぶ     | `POST /api/products/`   |
| Service    | ビジネスロジック（検証、計算など）            | 在庫が足りるか確認      |
| Model      | データベースとのやり取り（Django ORM）        | INSERT, SELECT 文の実行 |
| Serializer | データの検証と JSON 変換                      | ProductSerializer       |

> **Note**: Django では一般的に「Controller」ではなく「View」、「Entity」ではなく「Model」という用語を使います。これは Django が MTV（Model-Template-View）パターンを採用しているためです。

## 演習 4: 要件定義書の作成（15 分）

AI の出力を元に、要件定義書と設計書を Markdown ファイルとして保存します。

```sh
mkdir docs
```

- AI に要件定義書を生成させ、内容をレビュー・修正して `docs/requirements.md` として保存します。
- AI にシステム構成の設計書を生成させ、内容をレビュー・修正して `docs/design.md` として保存します。

完成したら、`Ans/` フォルダの完成例と見比べて確認してください。
